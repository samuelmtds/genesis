Instalação do exemplo useradmin

A instalação do exemplo useradmin é relativamente simples. A distribuição do exemplo useradmin se encontra nos pacotes de distribuição completa e da distribuição binária do genesis que podem ser encontrados aqui. A distribuição de todos exemplos do genesis já vem preparadas para rodar como aplicação independente (modo standalone/local) ou sobre o servidor de aplicações JBoss, que é um servidor gratuito e open-source. Isso não significa que o genesis só rode nesse servidor; o genesis foi desenvolvido totalmente compatível com as especificações J2EE.

Existem 4 passos simples para fazer a instalação do exemplo useradmin no servidor de aplicações:

  1. Baixe e instale o servidor de aplicações JBoss. Recomendamos utilizar a última das versões 3.2.x do JBoss.
  2. No diretório <DIRETÓRIO_DO_JBOSS>/server, faça uma cópia do diretório default para um novo diretório chamado useradmin. Dessa forma, teremos um diretório localizado em <DIRETÓRIO_DO_JBOSS>/server/useradmin que é uma cópia idêntica de <DIRETÓRIO_DO_JBOSS>/server/default
  3. Execute os scripts ant de deploy distribuídos juntamente com o exemplo (conforme instruções abaixo).
  4. Inicie o servidor JBoss (conforme instruções abaixo).

Após a instalação do JBoss e execução dos scripts de deploy, pode-se utilizar um link Java Web start ou os scripts de execução ant para rodar a aplicação exemplo (conforme instruções abaixo).

Propósito do exemplo useradmin

O exemplo useradmin tem como propósito mostrar ao usuário a maioria das funcionalidades que o projeto genesis oferece, e também demonstra a remotabilidade transparente, pois com o mesmo código-fonte é possível executar o exemplo utilizando tanto chamadas remotas como locais.

O exemplo possui dois diretórios de código fonte: o diretório <DISTRIBUIÇÃO_DO_GENESIS>/samples/useradmin/modules/client/src e o diretório <DISTRIBUIÇÃO_DO_GENESIS>/samples/useradmin/modules/shared/src que correspondem respectivamente ao módulo cliente da aplicação e ao módulo comum entre cliente e servidor.

No diretório de fontes do módulo cliente encontramos "trios" de arquivos fontes, que correspondem ao formulário da tela (form), arquivo XML da tela (XML do thinlet) e arquivo de visualização (view) da tela. Para exemplificar, consideremos a tela inicial do useradmin:

  • user-list.xml: arquivo XML de acordo com o padrão Thinlet que descreve como os componentes gráficos estão dispostos na tela. Os nomes dos componentes correspondem aos nomes de campos e métodos do formulário da tela (form).
  • UserListForm.java: classe Java cujos campos representam os dados da tela e que possui métodos que representam as ações da tela.
  • UserListView.java: classe Java responsável pela tela, onde é definido o tamanho da tela, qual é o arquivo xml da tela, além de fazer o binding de um form a tela, ou seja, indica qual form será utilizado para tratar os dados e eventos dessa tela. Outros métodos auxiliares podem existir nessa classe para possibilitar que essa tela possa abrir outras telas ou programaticamente executar ações dessa tela.

No diretório de fontes do módulo shared encontramos os arquivos fontes relativos a lógica de negócio (command) e os arquivos fontes relativos a entidades mapeadas em tabelas no banco de dados (databeans). Um exemplo de arquivo command é o arquivo net.java.dev.genesis.samples.useradmin.business.UserCreateCommand :

public class UserCreateCommand extends AbstractHibernateCommand {

   /**
    * @Transactional
    */
   public Serializable createUser(final User user) throws Exception {
      return getSession().save(user);
   }


}

Veja como os commands podem ser bem simples e objetivos. Esse command mostrado possui apenas um método transacional que salva um usuário no banco de dados. O commit e/ou rollback da transação é tratada automaticamente pelo genesis.

Como exemplo de um databean temos o net.java.dev.genesis.samples.useradmin.databeans.User que representa uma entidade Usuário que é mapeada para uma tabela utilizando tags do XDoclet:

/**
 * @hibernate.class
 *
 * @hibernate.query name="User.findByRole"
 *          query="from User u where u.role.code = :roleCode"
 */
public class User implements Serializable {
   private Long id;
   private String name;
   private String login;
   private String password;
   private String email;
   private Date birthday;
   private String address;
   private Role role;
   private Country country;
   private State state;

   /**
    * @hibernate.property
    */
   public Date getBirthday() {
      return birthday;
   }

   public void setBirthday(Date birthday) {
      this.birthday = birthday;
   }

   /**
    * @hibernate.property
    type="net.java.dev.genesis.samples.useradmin.databeans.CountryType"
    */
   public Country getCountry() {
      return country;
   }

   public void setCountry(Country country) {
      this.country = country;
   }

   /**
    * @hibernate.property
    */
   public String getEmail() {
      return email;
   }

   public void setEmail(String email) {
      this.email = email;
   }

   /**
    * @hibernate.property
    */
   public String getAddress() {
      return address;
   }

   public void setAddress(String address) {
      this.address = address;
   }

   /**
    * @hibernate.many-to-one
    */
   public Role getRole() {
      return role;
   }

   public void setRole(Role role) {
      this.role = role;
   }

   /**
    * @hibernate.id generator-class="increment"
    */
   public Long getId() {
      return id;
   }

   public void setId(Long id) {
      this.id = id;
   }

   /**
    * @hibernate.property
    */
   public String getLogin() {
      return login;
   }

   public void setLogin(String login) {
      this.login = login;
   }

   /**
    * @hibernate.property
    */
   public String getName() {
      return name;
   }

   public void setName(String name) {
      this.name = name;
   }

   /**
    * @hibernate.property
    */
   public String getPassword() {
      return password;
   }

   public void setPassword(String password) {
      this.password = password;
   }

   /**
    * @hibernate.property
    type="net.java.dev.genesis.samples.useradmin.databeans.StateType"
    */
   public State getState() {
      return state;
   }

   public void setState(State state) {
      this.state = state;
   }
}

O databean é uma classe com os getters e setters segundo a especificação de JavaBeans. Ele não precisa implementar nenhuma interface e nem estender nenhuma classe abstrata. Convém implementar java.io.Serializable para utilizá-lo como parâmetro e/ou retorno de chamadas remotas.

Execução do exemplo useradmin

Para executar esse exemplo, foram disponibilizados scripts ant para realizar esta tarefa. Para utilizar esses scripts, siga os passos abaixo:

  • Faça uma cópia do arquivo buid.properties.sample e modifique o nome da cópia para build.properties
  • Faça a edição do novo arquivo build.properties modificando ou comentado os valores das propriedades de acordo com o ambiente de execução
  • Na linha de comando, execute o script ant com o seguinte comando: ant (a task default, useradmin:all.with.webstart, será executada). Esse comando irá iniciar o processo de instalação do exemplo no JBoss. Esse processo será efetuado no diretório "useradmin" criado na etapa de instalação do JBoss anteriormente.
  • Uma vez finalizado o processo de instalação, o JBoss poderá ser iniciado através do comando: <DIRETÓRIO_DO_JBOSS>/bin/run -c useradmin
  • Após a completa inicialização do JBoss, o exemplo poderá ser executado através da URL http://localhost:8080/useradmin/. Através deste link, a aplicação exemplo será baixada, instalada e executada no computador do usuário, utilizando-se tecnologia Java Web start
  • Outra opção de execução do exemplo é através de uma task do ant: ant useradmin:run.remote .Este comando irá iniciar o exemplo com a utilização de chamadas remotas ao JBoss
  • Também é possível executar o exemplo apenas com chamadas locais, retirando-se a dependência do JBoss - para isso, basta utilizar o seguinte comando: ant useradmin:run.local

Através desse exemplo, mostramos como um projeto é estruturado e como usar as funcionalidades do genesis. É recomendado utilizar o projeto vazio para iniciar um projeto baseado no genesis e usar esse exemplo como uma referência.
