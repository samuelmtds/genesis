Conceitos básicos de componentes de interface gráfica

O modelo de componentes de interface gráfica do genesis é bastante expressivo embora simples e baseia-se em grande parte no uso de anotações. Estas anotações não são ainda interpretadas como metadados do Java 5.0, definidos pela JSR-175, mas são utilizados de maneira similar a tags do XDoclet. Os exemplos abaixo tornarão o uso claro.

Os seguintes conceitos e/ou características são abordados abaixo:

  • Configurando o genesis na inicialização
  • Tipos de componentes de interface gráfica
  • Entendendo as views
  • Utilizando as classes auxiliares de interface gráfica
  • Entendendo os forms
  • Implementando ações no form e na view

Configurando o genesis na inicialização

Diversos componentes precisam ser inicializados e/ou configurados para que o genesis funcione corretamente. Com intuito de facilitar esse processo, o genesis provê a classe net.java.dev.genesis.helpers.StartupHelper. Essa classe encapsula os passos que teriam de ser realizados manualmente, mas que tendem a ser bastante semelhantes para a maioria das aplicações.

Na maioria dos casos, basta adicionar a seguinte linha na inicialização da sua aplicação (provavelmente o método main(String[] args) de uma das classes):

new StartupHelper().initialize();

É importante salientar que esta classe precisa ser executada antes do uso de qualquer outra API do genesis pela sua aplicação em tempo de execução. O processo de inicialização executado por essa instrução, por padrão, consiste de:

  • Configuração do commons-beanutils, de modo a ser coerente com a API do genesis
  • Configuração do commons-jxpath, para uso de expressões condicionais
  • Registro de conversores e formatadores de dados padrão, a fim de fazer a integração entre os componentes Java e a API gráfica
  • Execução de uma simples classe de negócio vazia, net.java.dev.genesis.command.NoopCommand, em uma thread, apenas para inicializar os aspectos e conectar no servidor sem que o usuário tenha de esperar por isso.
  • Carga das regras de validação em uma thread, de modo que estas já estejam inicializadas quando forem necessárias.

Todo os pontos do processo são configuráveis através de métodos expostos em StartupHelper, de modo que a classe possa ser usada em quase todos os cenários. Por exemplo, caso uma aplicação não estiver usando comunicação com o servidor ou fizer isso logo na sua inicialização, pode-se desligar a execução de NoopCommand conforme mostrado abaixo:

StartupHelper helper = new StartupHelper();
helper.setRunNoopCommand(false);
helper.initialize();

StartupHelper registra um conversor e um formatador para java.util.Date de acordo com um formato de datas e java.util.Locale específicos. Por padrão, estes são a constante MM/dd/yyyy e o Locale padrão da JVM, mas existe uma outra versão do construtor que permite especificá-los diretamente conforme mostrado abaixo:

new StartupHelper(new Locale("pt", "BR"), "dd/MM/yyyy");

Por padrão, StartupHelper carrega as regras de validação. Caso a validação não esteja sendo utilizada, é necessário desligar este pré-carregamento conforme mostrado abaixo:

StartupHelper helper = new StartupHelper();
helper.setLoadValidatorRules(false);
helper.initialize();

Tipos de componentes de interface gráfica

A fim de atingir as metas de simplicidade e testabilidade, dois tipos de componente são combinados para formar a interface gráfica: uma View e um Form. Uma view é uma classe que realiza as operações específicas da API gráfica utilizada, enquanto o form é um simples JavaBean que contém a lógica de interface, é desacoplado da API gráfica e pode ser até mesmo reutilizado com outra API.

É importante salientar que o fato de o form estar desacoplado da API gráfica e ser apenas um JavaBean têm diversas conseqüências úteis para o desenvolvedor, como por exemplo:

  • Pode-se escrever testes unitários para a lógica de negócios.
  • O form pode estender qualquer classe e a herança pode ser utilizada para compartilhar características comuns.
  • A separação de código motiva o desenvolvedor a programar de forma mais clara e inteligível.
  • Será possível utilizar o mesmo form, sem modificações, numa interface Swing, por exemplo, tão logo o genesis suporte essa API

Essa arquitetura proposta, conforme mencionado no último item, permite que o mesmo modelo de form possa ser utilizado em diversas views. Atualmente o genesis suporta apenas Thinlet como modelo de implementação de views, mas versões futuras proverão suporte a Swing e até mesmo Thinlet em PDAs compatíveis com PersonalJava.

Entendendo as views

As views são fortemente acopladas com a API gráfica utilizada. O suporte a Thinlet hoje é implementado no genesis basicamente por duas classes em net.java.dev.genesis.ui.thinlet: BaseThinlet e sua subclasse BaseDialogThinlet.

BaseThinlet estende Thinlet das seguintes formas:

  • Expõe constantes para os nomes dos widgets (componentes gráficos) do Thinlet que são utilizados por toda API do Thinlet e também internamente
  • Provê diversos métodos set<Propriedade>(Object,<tipo>) e <tipo> get<Propriedade>(Object) para que as propriedades dos widgets possam ser manipuladas de modo fortemente tipado.
  • Implementa alguns métodos create<Widget>(..) que facilitam a geração dinâmica de componentes gráficos.
  • Possui métodos para exibir um bean na tela (displayBean(..) e populateFrom..(..)) bem como para capturar os valores exibidos em um bean (populate(Object)).
  • Possibilita o tratamento de erros através de handleException(Throwable) e métodos especializados como handleUIException(UIException), handleCustomException(Throwable), e handleUnknownException(Throwable).
  • Integra a validação e implementa o tratamento específico de erros para ela.
  • Implementa o binding (algo como junção) da view com o form através dos métodos bind(..), invokeFormAction(..) e outros.

Quando o modelo explicado neste documento é utilizado, i.e., o binding de view e form, somente as três últimas características são de interesse pelo código cliente. Por isso, somente estas serão exploradas neste documento.

BaseDialogThinlet é uma especialização de BaseThinlet que facilita a criação de janelas de diálogo. Logo todas as explicações referentes a BaseThinlet aplicam-se a ela também.

Configurando e exibindo uma janela

A criação da janela em que a view será exibida é de responsabilidade do programador. Uma maneira de criar a janela principal é utilizando a própria classe FrameLauncher que vem com o Thinlet. Abaixo, um exemplo de uso dessa técnica:

public class ExemploCriacaoFrame extends BaseThinlet {
   public ExemploCriacaoFrame() throws Exception {
      add(parse("exemplo_criacao_frame.xml"));
   }

   public static void main(String[] args) throws Exception {
      new FrameLauncher("Exemplo de criação de frame",
            new ExemploCriacaoFrame(), 800, 600);
   }
}

Este exemplo trivial pode ser suficiente para diversas situações, mas em algumas outras é necessário alterar as propriedades do frame antes de exibi-lo. Em casos em que seja necessário fazer as mesmas alterações para todos os frames na aplicação ou mesmo alterar configurações do Thinlet, recomenda-se a criação de uma superclasse intermediária. Esta técnica é utilizada nos exemplos providos com o genesis. A seguinte classe foi extraída do exemplo useradmin:

public abstract class BaseView extends BaseThinlet {
   public BaseView(String title, String xmlFile, int width, int height,
         boolean resizable) throws Exception {
      Frame f = new Frame();
      f.add(this);
      Insets is = f.getInsets();
      width += is.left + is.right;
      height += is.top + is.bottom;
      Dimension ss = getToolkit().getScreenSize();
      width = Math.min(width, ss.width);
      height = Math.min(height, ss.height);
      f.setBounds((ss.width - width) / 2, (ss.height - height) / 2, width,
            height);
      f.setResizable(resizable);
      f.setTitle(UIUtils.getInstance().getBundle().getString(title));
      f.addWindowListener(new WindowAdapter() {
         public void windowClosing(WindowEvent e) {
            System.exit(0);
         }
      });
      setAllI18n(true);
      setResourceBundle(UIUtils.getInstance().getBundle());
      add(parse(xmlFile));
   }
}

A classe mostrada acima cria seu próprio frame de modo a centralizá- lo, torná-lo redimensionável ou não, define o título através do ResourceBundle, o que permite internacionalização, configura a janela para encerrar a execução quando esta for fechada e configura a internacionalização da interface gráfica antes de adicionar o xml.

A classe BaseDialogThinlet possui construtores capazes de criar a Dialog - a partir de um Frame ou instância de BaseThinlet existentes ou criando um novo Frame - e outros que recebem uma instância criada externamente. Normalmente não será necessário configurar a instância de Dialog externamente. Por padrão é adicionado um listener que chama o método não-final onClose() quando a janela é fechada, permitindo que o comportamento em resposta a este evento possa ser customizado pelo desenvolvedor.

Assim como no caso do BaseThinlet, recomenda-se a criação de uma superclasse intermediária para o projeto. Um exemplo disso encontra-se na classe abaixo, também extraída do exemplo useradmin:

public abstract class BaseDialogView extends BaseDialogThinlet {
   public BaseDialogView(Frame frame, String title, String xmlFile,
         boolean resizable, boolean modal) throws Exception {
      super(frame);
      getDialog().setModal(modal);
      getDialog().setResizable(resizable);
      getDialog().setTitle(UIUtils.getInstance().getBundle().getString(title));
      setAllI18n(true);
      setResourceBundle(UIUtils.getInstance().getBundle());
      add(parse(xmlFile));
   }
}

O código é bastante semelhante ao anterior, apenas configurando a instância de Dialog - que pode ser obtida através do método getDialog(), como mostrado acima - para ser modal (bloquear o acesso a outras janelas) ou não e redimensionável ou não.

Utilizando as classes auxiliares de interface gráfica

Algumas classes auxiliares são providas juntamente com o genesis-client para acelerar o processo desenvolvimento da interface gráfica. Elas implementam tarefas comuns, como a exibição de mensagens, confirmações por parte do usuário e exibição de exceções. Todas essas classes encontram-se no pacote net.java.dev.genesis.ui.thinlet.

MessageDialog

A primeira dessas classes auxiliares é MessageDialog, que exibe uma mensagem para o usuário em uma Dialog com um botão "Fechar". Dois métodos estáticos show(..) são providos com o objetivo de tornar o uso desta classe ainda mais prático. Ambos aceitam o título da janela como segundo parâmetro e a mensagem como terceiro, sendo que o primeiro argumento pode ser uma instância de Frame ou de BaseThinlet. A segunda versão torna seu uso bastante prático dentro de uma classe que estenda BaseThinlet como mostra o código abaixo:

public void exibirMensagem() {
   MessageDialog.show(this, "Título", "Mensagem a ser exibida");
}

OptionDialog

Para se exibir uma escolha para o usuário pode-se utilizar a classe OptionDialog. São providas, de forma semelhante a MessageDialog, duas versões do método display(..), aceitando uma instância de BaseThinlet ou de Frame, o título da janela e a pergunta a ser exibida. Ao utilizar algum desses métodos é exibida a janela de diálogo com três botões (Sim/Não/Cancelar). Existe um método displayYesNo(BaseThinlet,String,String) para casos em que haja somente duas opções. Todos os três métodos retornam um dos valores da enumeração Option, de modo que pode-se saber que opção o usuário escolheu. O uso de OptionDialog em uma classe que estenda BaseThinlet é mostrado abaixo:

public void fazOperacao() {
   Option option = OptionDialog.displayYesNo(this, "Título", "Deseja " +
         "realmente fazer isso?");

   if (option == Option.YES) {
      // faz a operacao
   }
}

ErrorReporterDialog

A classe ErrorReporterDialog permite exibir uma exceção para o usuário com uma mensagem descritiva além de todo o stack trace. Assim como no caso de MessageDialog existem também duas versões do método show(..): uma para BaseThinlet e outra para Frame. O exemplo abaixo mostra como utilizar esta classe em uma subclasse de BaseThinlet:

public void exibeErro(Throwable t) {
   ErrorReporterDialog.show(this, "Erro", t.getMessage(), t);
}

O uso direto de ErrorReporterDialog diretamente no código das views provavelmente será desnecessário, visto que o tratamento de erros padrão implementado em BaseThinlet já utiliza esta classe, conforme explicado posteriormente.

Entendendo os forms

Os forms contêm a lógica da interface gráfica e são desacoplados da API gráfica utilizada para exibir as telas. Um form é apenas um JavaBean que possui anotações que indicam como suas propriedades e métodos devem ser tratados pela interface gráfica.

Visto que o form não está diretamente ligado a API gráfica, é necessário fazer a ligação entre ele e esta API. Esta ligação é chamada de binding e cada especialização do genesis-client possuirá uma maneira de fazer isto. Utilizando a única implementação suportada no momento, a de Thinlet, isso é feito através do método bind(Object), que liga dinamicamente o form à view.

Um exemplo muito simples de uma interface gráfica completa pode ser observado abaixo. Temos todos os componentes necessários para implementar uma simples tela de login. Primeiramente, vemos o xml da tela, contido num arquivo de nome login.xml:


 1. <panel gap="5" top="5" left="5" bottom="5" right="5" weightx="1"
 2.        weighty="1">
 3.    <panel gap="5" columns="2" weightx="1">
 4.       <label text="Usuario" font="bold" />
 5.       <textfield name="usuario" columns="10" />
 6.       <label text="Senha" font="bold" />
 7.       <passwordfield name="senha" columns="10" />
 8.    </panel>
 9.    <panel gap="5" halign="center" weightx="1">
10.       <button name="login" text="Login" />
11.       <button name="limpar" text="Limpar" />
12.    </panel>
13. </panel>

A seguir, vemos o código da view:

public class LoginView extends BaseDialogThinlet {
   public LoginView() throws Exception {
      getDialog().setTitle("Login");

      add(parse("login.xml"));
      bind(new LoginForm());
   }

   public static void main(String[] args) throws Exception {
      new LoginView().display();
   }
}

Vemos no código acima o uso do método bind(Object) passando o form em questão. Abaixo, temos o código do form utilizado neste exemplo:

Através deste exemplo podemos observar algumas coisas importantes sobre a definição de um form. Um form recebe a anotação @Form na classe, conforme observado nas linhas 1-3 do exemplo. Esta anotação é que indica ao genesis que esta classe deve ser tratada como form, o que requer um tratamento especial através de um aspecto do genesis-aspect.

As propriedades expostas através de setters e getters - neste caso, usuario (linhas 8-14) e senha (linhas 16-22) - podem ser "ligadas" dinamicamente a diversos widgets da interface gráfica, como textfields, passwordfields, labels e outros. Para que isto aconteça o name dos widgets no xml (linhas 5 e 7 do xml) tem de ser iguais aos nomes das propriedades. Tanto o valor da propriedade no form quanto sua exibição na tela estarão sincronizados quando o componente que representa a propriedade perder o foco. No caso do exemplo dado, isso acontece quando o usuário pressiona a tecla TAB ou clica com o mouse em algum outro ponto da tela quando o foco está nas caixas de texto usuario ou senha.

Caso se deseje que determina propriedade não possa ser atualizada nem exibida pelo genesis, deve-se anotar o método getter da propriedade com a anotação @NotBound.

Os métodos também podem ser ligados a botões na interface gráfica. Para isso, têm de receber a anotação @Action (linhas 24-26 e 32-34) e devem existir botões com valor da propriedade name igual ao deles na interface gráfica (linhas 10-11 do xml). Assim, quando o botão é acionado o método com nome equivalente é invocado no form. No exemplo dado, ao clicar em Login o valor dos campos Usuário e Senha é impresso na saída padrão enquanto acionar o botão Limpar exclui o conteúdo dos campos tanto na tela quanto no form.

Implementando ações no form e na view

Embora o genesis possibilite que seja feito o binding de ações do form com controles na tela, nada impede o uso do recurso já presente no Thinlet de se invocar métodos configurando-os no xml da tela. Desta forma, uma questão recorrente na mente dos desenvolvedores é quando usar @Action e quando utilizar action="metodo()" diretamente na classe da view. Alguns princípios básicos podem ser usados para se tomar esta decisão.

Se uma ação modifica o form ou não está ligada a API gráfica, esta deve ser implementada no form com o uso de @Action. Se a resposta a determinado controle é simplesmente exibir uma nova tela, janela de diálogo ou alguma tarefa específica da API gráfica sendo utilizada, deve- se implementá-la na view e acioná-la com action="metodo()". Porém, existem situações não tão simples e outras nas quais a manipulação do form e da view se misturam. Estes casos são considerados abaixo.

Usando @BeforeAction

Consideremos o seguinte exemplo: deve-se implementar a exclusão de um dado. Mas antes de exclui-lo o usuário deve confirmar a operação. Essa confirmação poderia ser implementada exibindo uma OptionDialog para o usuário e prosseguindo com a operação somente se ele concordasse. A operação de exclusão não é específica da API gráfica e, por tanto, deve ser implementada no form. Como implementar isso?

A fim de se executar um método na interface gráfica antes de invocar um método no form - e até mesmo impedir a execução do mesmo - utiliza-se a anotação @BeforeAction. Esta anotação deve ser colocada no método da view que se deseja executar antes do método do form e pode tanto retornar boolean como void. Quando o método retorna false o método no form não é invocado.

Se o nome do método na view for o mesmo do método no form não é necessário utilizar nenhum parâmetro adicional para a anotação. Em certos casos, porém, o nome do método na view não pode ser o mesmo utilizado no form. Para indicar a execução de qual método do form deve ser precedida da execução do método na view, coloca-se ao lado da anotação o nome do método no form. Um exemplo completo de uso encontra- se abaixo. Primeiro, a view:

public class ExemploBeforeActionView extends BaseThinlet {
   public ExemploBeforeActionView() {
      // lógica de inicializacao
      bind(new ExemploBeforeActionForm());
   }

   /**
    * @BeforeAction excluir
    */
   public boolean confirmaExclusao() {
      return OptionDialog.displayYesNo(this, "Excluindo registro...",
            "Deseja realmente excluir o registro?") == Option.YES;
   }

   // ...
}

Abaixo temos o form:

/**
 * @Form
 */
public class ExemploBeforeActionForm {
   /**
    * @Action
    */
   public void excluir() {
      // implementa aqui a lógica de exclusão.
   }

   // ...
}

Quando o botão com name="excluir" é acionado, o genesis invoca o método confirmaExclusao() em ExemploBeforeActionView visto que este está configurado com a anotação @BeforeAction excluir. Caso o nome do método na view fosse excluir não seria necessário passar o nome do método como parâmetro para a anotação.

Visto que confirmaExclusao() possui como retorno um boolean, o método no form só será invocado quando o retorno for true. No exemplo específico, isso só acontecerá quando o usuário escolher a opção "Sim" na janela que será exibida. Quando a confirmação ocorre, o método excluir() é invocado pelo framework e o tratamento do acionamento do botão, encerrado.

Usando @AfterAction

Após a execução de um método do form pode ser necessário utilizar algum recurso da API gráfica, como, por exemplo, fechar a janela sendo exibida. A fim de possibilitar isso o genesis suporta a anotação @AfterAction, que deve ser colocada no método da view a ser invocado. Sua sintaxe é muito similar a da anotação @BeforeAction, exceto que o método que possui a anotação deve retornar void. Abaixo há um exemplo do uso:

public class ExemploAfterActionView extends BaseDialogThinlet {
   public ExemploAfterActionView() {
      // lógica de inicializacao
      bind(new ExemploAfterActionForm());
   }

   /**
    * @AfterAction salvar
    */
   public void fechaJanela() {
      getDialog().dispose();
   }

   // ...
}

/**
 * @Form
 */
public class ExemploAfterActionForm {
   /**
    * @Action
    */
   public void salvar() {
      // implementa aqui a lógica para salvar
   }

   // ...
}

No exemplo acima, quando o botão com name="salvar" é chamado, o método salvar() é invocado no form. Após a execução, o método fechaJanela() na view, que está anotado como @AfterAction salvar é executado. Assim como no caso de @BeforeAction, caso o nome do método na view também fosse salvar() não seria necessário passá-lo na anotação.

Usando refreshView(..) e refreshViewFromForm(Object)

Algumas vezes os valores de um form podem ser atualizados externamente ou diretamente pela view - para coordenar a iteração de diversos forms, por exemplo. Contudo, visto que estas alterações não foram gerenciadas pelo genesis, tanto o cache interno do framework como as informações exibidas ficam fora de sincronia, causando uma série de problemas. Os métodos refreshView(), refreshView(Object) e refreshViewFromForm(Object) podem ser utilizados nestes cenários.

O primeiro método, refreshView(), deve ser utilizado quando somente um form está associado a view. refreshView(Object) recebe como parâmetro o widget ao qual um ou vários forms estão ligados e processa quaisquer atualizações encontradas neles. refreshViewFromForm(Object) atualiza o widget ligado ao form passado como parâmetro. Um exemplo de uso do primeiro método pode ser visto abaixo:

public class ExemploRefreshViewView extends BaseThinlet {
   private ExemploRefreshViewForm form;

   public ExemploRefreshViewView() {
      // lógica de inicializacao
      bind(form = new ExemploRefreshViewForm());
   }

   public void selecionaEntidadeRelacionada() {
      // exibe uma tela auxiliar de selecao de entidade e retorna
      Entidade entidade = new ConsultaEntidadeView().getEntidade();

      if (entidade != null) {
         form.setEntidade(entidade);
         refreshView();
      }
   }

   // ...
}

/**
 * @Form
 */
public class ExemploRefreshViewForm {
   private Entidade entidade;
   // ...

   public Entidade getEntidade() {
      return entidade;
   }

   public void setEntidade(Entidade entidade) {
      this.entidade = entidade;
   }

   // ...
}

Usando invokeFormAction(..)

Em algumas ocasiões pode ser necessário invocar uma ou mais ações do form de dentro de um método da view sob determinadas condições específicas e o uso de @BeforeAction e @AfterAction não é possível. Embora a execução direta de um método do form seja possível, ela ocorrerá sem que o framework seja notificado disso, o que terá diversas conseqüências como a não atualização da interface gráfica, por exemplo.

Para chamar um método do form e manter o sincronismo automático deve- se utilizar os métodos invokeFormAction(String) ou invokeFormAction(Object,String). O parâmetro String que ambos recebem é o nome do método do form a ser invocado. A segunda versão do método aceita a instância do form em que o método deve ser invocado e é útil quando vários forms estão ligados a view. Um exemplo de uso da primeira versão do método encontra-se abaixo:
